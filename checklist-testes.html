🚀 Cronograma — Backup, Homologação e Deploy Controlado

Planejamento em 3 frentes independentes que evoluem em paralelo

🗄️ FRENTE 1 — Backup & Recuperação

Semana 1 — Imediato (já disponível)

Backup automático do Supabase

O Supabase Pro já inclui backup diário automático. Ative e configure:

  • Supabase Dashboard → Project Settings → Database → Backups → verificar retenção (7 dias no free, 30 dias no Pro)
  • Testar restore: baixar backup mais recente → restaurar em projeto de teste → confirmar integridade
  • Anotar o processo de restore em documento interno (runbook)
Semana 2 — Backup de código

Repositório Git + exportação Supabase

  • Confirmar que o repositório está no GitHub (código já versionado) ✅
  • Adicionar script npm run backup:db que exporta schema + dados via pg_dump para arquivo local
  • Agendar exportação semanal via GitHub Actions (cron) → armazenar no repositório privado (branch backups)
  • Backup das variáveis de ambiente: manter cópia criptografada em 1Password/Bitwarden
Semana 3-4 — Monitoramento e alerta

Alerta de falha + teste mensal de restore

  • Configurar uptime monitor gratuito (Better Uptime ou UptimeRobot) → alerta WhatsApp/email se o sistema cair
  • Criar checklist mensal: testar restore do backup em projeto Supabase temporário
  • Documentar RTO (Recovery Time Objective): meta <2h para restaurar o sistema

🧪 FRENTE 2 — Ambiente de Homologação (Staging)

Semana 1-2 — Criar ambiente staging

Estrutura: branch + projeto Supabase separado

  • Criar branch develop no Git → todo desenvolvimento novo vai para esta branch
  • Criar projeto Supabase separado chamado gestescolar-staging (plano free)
  • Criar projeto Vercel separado apontando para a branch develop → URL automática: gestescolar-staging.vercel.app
  • Configurar variáveis de ambiente do staging: apontar para Supabase staging + Asaas sandbox
  • Resultado: cada push na develop faz deploy automático no staging
Semana 2-3 — Workflow de trabalho

Fluxo: develop → staging → main (produção)

  • Desenvolvimento: criar branch feature/nome-da-funcionalidade a partir de develop
  • Teste local: testar na máquina com vercel dev
  • Pull Request: abrir PR de feature/X para develop → Vercel cria preview URL automaticamente
  • Homologação: testar na preview URL usando o checklist deste arquivo
  • Merge para main: somente após testes aprovados → deploy automático em produção
Semana 3-4 — Dados de teste

Seed de dados para homologação

  • Criar script supabase/seed.sql com dados fictícios: 2 escolas, 10 alunos, 3 professores, faturas variadas
  • Script de reset: limpar staging e reseeder antes de cada ciclo de teste
  • Nunca usar dados reais no staging — criar dados sintéticos

🚢 FRENTE 3 — Deploy Controlado para Produção

Como subir SOMENTE o que foi testado

Regra de ouro: nada vai para main sem PR + checklist

  • Proibir push direto na branch main → configurar "Branch Protection Rule" no GitHub: require pull request before merging
  • PR deve ter pelo menos 1 aprovação (mesmo que seja você mesmo — obriga a revisar)
  • Checklist no template do PR: "Testei no staging? Funcionalidade X funciona? Não quebrei Y?"
  • Vercel mostra o diff do deploy → comparar antes de promover para produção
Deploy parcial — subir 1 funcionalidade por vez

Feature flags simples (sem biblioteca externa)

  • Para features instáveis, adicionar flag na tabela platform_settings: group=feature, key=nova_funcionalidade, value=false
  • No código verificar: if (featureFlag === 'true') { ... }
  • Deploy vai para produção com feature OFF → ativar via superadmin quando pronto → sem novo deploy
  • Em caso de bug: desativar a flag em segundos sem rollback de código
Rollback de emergência

Reverter deploy em <1 minuto pelo Vercel

  • Vercel guarda histórico de todos os deploys → Vercel Dashboard → Deployments → clicar no deploy anterior → "Promote to Production"
  • Para banco: se migration errada → reverter com script de rollback (manter sempre script down para cada migration)
  • Nunca fazer migration destrutiva (DROP, RENAME column) sem backup manual primeiro

📋 Resumo: Estrutura Final de Branches

main (produção — gestescolar.com.br)
  ↑ merge via PR aprovado após homologação
develop (staging — gestescolar-staging.vercel.app)
  ↑ merge via PR após teste local
feature/nome-da-feature (desenvolvimento local)
  → vercel dev (teste na máquina)
✅ Cada funcionalidade vive em sua própria branch
✅ Staging usa Supabase e Asaas separados (nunca dados reais)
✅ Produção só recebe código testado e aprovado
✅ Rollback em 1 clique no painel Vercel
