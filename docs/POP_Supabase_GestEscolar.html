5. Estrutura das Tabelas

📋 schools — Escolas

  • id (UUID) — Identificador único
  • name — Nome da escola
  • cnpj — CNPJ
  • email, phone, address — Contato
  • plan_id — Plano contratado (free, gestao_100, etc)
  • pix_key — Chave PIX para recebimentos

👤 users — Usuários

  • auth_id — Vínculo com Supabase Auth (login real)
  • school_id — Escola vinculada
  • name, email, cpf, phone — Dados pessoais
  • role — Função (gestor, professor, pai, financeiro, etc)
  • active — Se o usuário está ativo

🎓 students — Alunos

  • matricula — Matrícula gerada automaticamente
  • class_id — Turma vinculada
  • status — ativo/inativo
  • monthly_fee — Valor da mensalidade
  • responsaveis (JSON) — Lista de responsáveis
  • parent_id — Vínculo com usuário pai

💰 invoices — Mensalidades

  • student_id, student_name — Aluno
  • description, amount, due_date — Detalhes
  • status — pendente/pago/vencido/cancelado
  • asaas_id — ID da cobrança no Asaas (futuro)

📨 messages — Mensagens

  • from_user_id, to_user_id — Remetente e destinatário
  • student_id — Aluno relacionado
  • subject, text — Conteúdo
  • read — Se a mensagem foi lida

6. Segurança — Row Level Security (RLS)

O schema já configura automaticamente o Row Level Security (RLS) em todas as tabelas. Isso significa que:

  • ✅ Cada escola só vê seus próprios dados
  • ✅ Um usuário da Escola A NÃO consegue acessar dados da Escola B
  • ✅ O superadmin consegue ver dados de todas as escolas
  • ✅ A segurança é aplicada no nível do banco de dados (não depende do frontend)
⚠️ NUNCA desabilite o RLS nas tabelas. Ele é essencial para a segurança do sistema multi-tenant. Se uma tabela aparecer com RLS desabilitado, me avise imediatamente.
